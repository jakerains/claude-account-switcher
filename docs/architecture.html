Claude Account Switcher

One Claude experience. Multiple usage accounts.

The switcher gives every alias the same MCPs, skills, plugins, sessions, resumes, history, settings, and project state. The only thing that changes is the Claude account credential used for subscription usage. Normal agent runs route directly to Claude, so there is no proxy in the hot path.

Architecture

Generated aliases do not replace Claude Code. Normal commands set CLAUDE_CONFIG_DIR and launch the real Claude binary directly. MCP commands use the managed sync path so shared MCP config and OAuth stay in sync.

Command lifecycle

Daily agent work takes the direct route. MCP commands take the managed route because they may change shared MCP authorization.

1

Alias captures intent

The shell function checks the first argument. Most commands skip the manager entirely.

2

Normal runs go direct

CLAUDE_CONFIG_DIR=<profile> claude "$@" launches the real Claude binary.

3

MCP runs sync

mcp commands use cca run-managed to merge MCP OAuth and config.

4

Usage stays isolated

Claude account credentials stay in the selected profile while shared state remains common.

State split

The product is intentionally strict about what is private. MCP authorization is shared; Claude subscription identity is not.

Shared across every profile

  • MCP server config and MCP OAuth
  • Skills, plugins, commands, agents, and hooks
  • Settings, projects, tasks, sessions, resumes, and history
  • Local MCP approvals and health caches

Private to one profile

  • Claude/Anthropic OAuth credential
  • Claude account identity fields
  • Subscription and usage-plan metadata
  • Account-specific access caches
